## 二、短期（1-2 周）—— 补齐工程基础

### S1. Playwright async_api 改造（根治 P0-4）
**目标**：消除多线程并发崩溃，提升爬虫吞吐量。

**方案**：
```
core/async_scraper.py        — 新建，async_api 单事件循环
  ├─ AsyncBrowserPool        — 单 playwright 实例 + 多 context
  ├─ async search(platform)  — 协程化搜索
  └─ run_until_complete()    — 统一调度
core/monitor.py              — _check_all_concurrent 改用 asyncio.gather
datasources/ctrip_browser_source.py — 保留 sync 版作降级
```
- monitor 主线程跑事件循环，ThreadPoolExecutor 仅用于 DB 写（sqlite 线程安全）
- 预期收益：4 并发搜索从"偶发崩溃"到稳定，吞吐 +50%

### S2. 测试体系搭建
**目标**：建立可回归的测试基线。

**方案**：
```
tests/
├── conftest.py              — pytest fixtures（临时 DB、mock app、test_client）
├── unit/
│   ├── test_database.py     — CRUD + rollback + close_all + 并发写
│   ├── test_aggregator.py   — 去重 + 回填 + 跨平台合成
│   ├── test_notifier.py     — 去重 + 脱敏 + 渠道分发
│   └── test_prediction.py   — _stable_seed 确定性 + 6 profile 边界
├── integration/
│   ├── test_api_routes.py   — 23 端点全覆盖（输入校验/状态码/CORS/限流）
│   └── test_monitor.py      — start/stop 并发 + price None 过滤
└── tools/verify_fixes.py    — 已有，保留作冒烟检查
```
- 目标覆盖率：core/ 70%+，api/ 80%+
- CI：GitHub Actions（如推送 GitHub）或本地 pre-commit hook

### S3. config.py 拆分
**目标**：550 行配置 → 数据与逻辑分离。

**方案**：
```
config/
├── __init__.py              — 聚合导出（保持 `from config import X` 兼容）
├── settings.py              — 环境变量、路径、间隔（~50 行）
├── data/
│   ├── cities.json          — 150 城市 + CITY_CODES + CITY_GROUPS
│   ├── routes.json          — 56 热门航线 + ROUTE_AIRLINES
│   ├── platforms.json       — 28 购买平台 + AIRLINE_OFFICIAL_SITES
│   └── airlines.json        — DOMESTIC/INTERNATIONAL_AIRLINES
└── loader.py                — JSON 懒加载 + 热重载支持
```
- 城市航线数据可后续接外部 API（如携程城市码接口）动态更新

### S4. 真实数据源降级链
**目标**：ctrip 失败时优雅降级，不留空数据。

**方案**：
```
core/source_chain.py         — 新建
  ├─ SourceChain([ctrip_browser, ctrip_api, skyscanner, mock])
  ├─ search(query) → 按优先级尝试，首个成功即返回
  ├─ 失败计数 → 自动熔断（连续 3 次失败暂停 10 分钟）
  └─ 健康检查 → /api/sources/status 展示各源可用性
```
- 前端展示数据来源标记（真实 / mock / 降级）

### S5. 结构化日志
**目标**：可追踪、可聚合的日志。

**方案**：
- `structlog` 或标准 `logging` + JSON formatter
- 每个请求带 `request_id`（uuid4 短码），贯穿 API → monitor → notifier
- 日志文件按天滚动 + 关键错误 webhook 推送

## 三、中期（1-2 月）—— 架构升级

### M1. 前端框架化（Vue 3 + Vite）
**目标**：替换 989 行原生 JS，降低维护成本。

**方案**：
```
frontend/
├── src/
│   ├── App.vue
│   ├── views/
│   │   ├── Dashboard.vue    — 总览（监控数 / 告警 / 航线最低价）
│   │   ├── QueryList.vue    — 查询管理
│   │   ├── PriceChart.vue   — 价格历史 + 预测
│   │   ├── Compare.vue      — 多平台比价
│   │   └── Alerts.vue       — 告警配置 + 历史
│   ├── components/
│   │   ├── FlightCard.vue
│   │   ├── PriceChart.vue   — 封装 Chart.js
│   │   └── PlatformBadge.vue
│   ├── api/                 — axios 封装
│   └── stores/              — Pinia 状态管理
├── vite.config.ts
└── package.json
```
- 渐进迁移：先包装现有 app.js 为 Vue 组件，再逐页重写
- 构建产物 `dist/` 由 Flask 静态服务

### M2. 任务队列（RQ 或 Celery）
**目标**：替换 ThreadPoolExecutor，爬虫任务可观测、可重试。

**方案**：
```
core/tasks.py
  ├─ @job search_query(query_id)    — 单查询搜索
  ├─ @job check_all_monitors()      — 轮询所有监控
  ├─ @job train_prediction(qid)     — 模型训练
  └─ @job send_notification(...)    — 通知发送
```
- Redis 作 broker（轻量，单机即可）
- Web 端 `/api/jobs` 查看任务状态
- 失败自动重试 3 次 + 死信队列

### M3. 数据源插件化
**目标**：新增数据源零改动核心逻辑。

**方案**：
```
datasources/registry.py      — 新建
  ├─ REGISTRY = {}           — name → SourceClass
  ├─ @register("ctrip")      — 装饰器注册
  ├─ get_available_sources() — 按 is_available() 过滤
  └─ load_from_config()      — 从 config 动态加载
```
- 每个数据源实现 `BaseDataSource`（已有）+ `close()`（待加）+ `health_check()`
- 配置文件指定启用哪些源，无需改 monitor.py

### M4. 预测模型持久化
**目标**：模型训练一次，多次复用；支持版本对比。

**方案**：
```
core/model_store.py          — 新建
  ├─ save_model(qid, model, version)  → models/{qid}_v{ver}.pkl
  ├─ load_latest(qid)
  ├─ list_versions(qid)
  └─ compare_versions(v1, v2)         — MAPE/RMSE 对比
```
- 训练任务定时跑（如每天凌晨），不阻塞查询
- `/api/queries/<id>/model-info` 展示当前模型版本 + 训练数据量 + 误差

### M5. 容器化部署
**目标**：一键部署，环境一致。

**方案**：
```dockerfile
# Dockerfile
FROM python:3.13-slim
RUN apt-get update && apt-get install -y chromium
COPY . /app
RUN pip install -r requirements.txt
CMD ["python", "main.py"]
```
```yaml
# docker-compose.yml
services:
  app:        — Flask + monitor
  redis:      — 任务队列
  nginx:      — 反代 + 静态资源
volumes:
  db_data:    — SQLite 持久化
```

## 四、长期（3-6 月）—— 功能扩展

### L1. 多用户系统
- **认证**：JWT + Flask-Login，注册/登录/找回密码
- **权限**：每用户独立查询/告警，管理员可看全部
- **个人看板**：我的航线 / 我的告警 / 价格追踪历史
- **数据库**：SQLite → PostgreSQL（多用户并发写）

### L2. 智能告警 2.0
- **预测联动**：告警不只是"跌破目标价"，而是"预测 7 天内最低 ¥520，当前 ¥580，建议等 3 天"
- **时机推荐**：基于历史 88 天 pattern + 节假日，给出"最佳购买日期"
- **多条件告警**：价格区间 / 涨跌幅 / 航班时刻 / 舱位 / 经停数
- **告警渠道扩展**：Telegram / 钉钉 / 短信

### L3. 移动端
- **PWA**：Vue 3 + Vite PWA 插件，可安装到手机桌面，离线查看历史
- **微信小程序**（可选）：复用 API，小程序前端 + 订阅消息推送

### L4. 数据仓库与分析
- **历史价格仓库**：price_records 分区表（按月），保留 2+ 年
- **航线热度排行**：基于查询频率 + 价格波动
- **价格指数**：类似 CPI 的"航线价格指数"，按周/月发布
- **可视化大屏**：热门航线地图 + 实时价格热力图

### L5. 开放 API 与生态
- **API 文档**：FastAPI 自动生成 OpenAPI，或 Flask + flasgger
- **第三方集成**：Webhook 回调（价格变化推送）、RSS 订阅、IFTTT/Zapier
- **SDK**：Python/JS SDK，方便他人二次开发

## 五、技术栈演进建议

| 层 | 当前 | 建议 | 迁移时机 |
|----|------|------|----------|
| Web 框架 | Flask (sync) | **保留 Flask** + ASGI (uvicorn) | 单用户够用；多用户时考虑 FastAPI |
| 任务队列 | ThreadPoolExecutor | **RQ**（轻量）| M2 阶段 |
| 前端 | 原生 JS | **Vue 3 + Vite + Pinia** | M1 阶段 |
| 数据库 | SQLite (WAL) | **保留 SQLite** → PostgreSQL | L1 多用户时 |
| 爬虫 | Playwright sync | **Playwright async** | S1 阶段（优先） |
| 部署 | bat 脚本 | **Docker + docker-compose** | M5 阶段 |
| 日志 | logging | **structlog** + JSON | S5 阶段 |
| 测试 | 无 | **pytest + pytest-asyncio** | S2 阶段 |
| 配置 | config.py 硬编码 | **JSON 数据文件 + env** | S3 阶段 |

### 不建议引入的
- **Django**：过重，Flask 已够用
- **Celery**（除非任务量很大）：RQ 更轻，单机够用
- **React**（本项目）：Vue 上手快，单人维护友好；React 生态大但本项目用不到
- **GraphQL**：REST 23 端点够清晰，GraphQL 增加复杂度无收益

## 七、一句话总结

当前项目"能跑、能用、架构清晰"，但"爬虫不稳、测试缺失、前端难维护"是三大瓶颈。短期补 async 爬虫 + 测试 + 降级链可让"真实数据可用"从偶发变为稳定；中期前端框架化 + 任务队列让项目"可维护、可扩展"；长期多用户 + 智能告警 + 移动端让工具变成"产品"。
